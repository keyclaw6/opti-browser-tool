# Opti Browser Tool foundations and bring-up audit

## 1. Scope verdict

**Foundations coherent: `CONDITIONAL`. Current foundation readiness: `R1 scaffold`.**

The project objective is coherent: optimize reliable, repeatable browser-task completion, transfer, and recovery around a mostly fixed executor; optimize efficiency only among comparably reliable systems. Success means repeatedly producing trace-explained improvements that transfer, survive repetitions and regressions, do not exploit evaluators, and respect platform constraints (`PROJECT_CHARTER.md:5-17`, `PROJECT_CHARTER.md:55-57`, `docs/adr/0001-project-constitution.md:14-28`).

The condition is material. The task inventory and orchestration shells are unusually well specified for a scaffold, but the repository cannot yet produce trustworthy real benchmark evidence. A generic external command can report `passed` without verifier, trace, artifact, or full run identity and be labeled benchmark-reportable; the loop also treats absent or malformed traces as no signal rather than invalid evidence. Separately, the five environments, five bridges, canonical trace writer/store, exact model invocations, per-task verifier admissions, calibration, hidden holdout, real noise band, real Analyst, and seed browser harness do not exist as an executable vertical slice.

Today this is a **research specification plus orchestration scaffold**, not a bring-up-ready platform or real evaluation system. The repository says the harness contains no browser code (`harness/README.md:1-5`), the bridge registry is disabled and points at absent scripts (`evals/config.example.json:3-33`), and the source audit explicitly did not launch browsers, reset environments, or run evaluators (`research/benchmarks/task-candidates/batch-1-source-audit.md:3-6`). Fixtures prove plumbing only.

The project can advance to `R2 bring-up-ready` after the two P0 evidence-boundary defects are closed and a clean-host, one-real-task-per-source smoke demonstrates deterministic reset, strict identity, native verification, canonical trace/artifact capture, and admitted verifiers. `R3 real calibration-ready` additionally requires materialized/admitted tasks, a preregistered sampling rule, exact model/seed identity, a private holdout source, and measured real noise/judge evidence.

### Evidence-state vocabulary

- `verified-real`: reproduced against the current host or real upstream/runtime state.
- `implemented-unverified`: executable code exists, but not verified against real benchmark infrastructure.
- `simulated-only`: fixture or synthetic path only.
- `documented-only`: prose/schema contract exists without its producer.
- `missing`: required implementation/evidence is absent.
- `externally-blocked`: depends on credentials, gated service, assets, or unavailable upstream state.
- `contradicted`: active artifacts make incompatible claims.

### Ordering and circularity defects

The runbook freezes the suite using only the frontier reference, although the task report requires both a simple baseline and the chosen strong reference (`research/benchmarks/task-candidates/batch-1-report.md:122-136`). It then measures MiniMax noise in B5 before B7 creates the MiniMax seed harness (`docs/BRINGUP_RUNBOOK.md:92-129`). Noise measures stochasticity for a fixed identity; it does not establish that the frozen suite has a useful difficulty distribution for MiniMax. The shortest correction is to resolve/build the seed early enough to run a MiniMax baseline before suite freeze, then measure noise only after the suite and exact seed identity are frozen.

The holdout step is also circular: all 140 public candidates must burn as visible calibration tasks, yet the runbook says to “carve” a hidden holdout from the calibration phase. ADR-0015 correctly says the holdout cannot live in the public repository (`docs/adr/0015-auto-research-loop-architecture.md:108-115`); a separate private source must exist before optimizer exposure.

## 6. Task/suite findings

### What is verified

- Raw, canonical normalized, index, and 140 by-ID records have identical ID sets. Counts are exactly `30 REAL / 30 WebArena-Verified / 30 WorkArena++ L2 / 30 VisualWebArena / 20 WARC-Bench`; the repository documents the same split (`docs/TASK_DATA_GUIDE.md:102-109`).
- Every normalized row has provenance and required upstream locator fields; source/native-ID/seed tuples are unique. The lock records versions, manifest locators, source hashes, and package hashes (`research/benchmarks/task-candidates/batch-1-sources.lock.json:5-85`). Those hashes are internally consistent with the catalog, but upstream bytes were not re-fetched in this audit.
- `candidate-pool.json` and provisional `primary.json` have the same ordered 140 IDs. `smoke.json` has 20 unique IDs and is a strict subset. `regression.json` currently equals smoke and is explicitly provisional (`docs/TASK_DATA_GUIDE.md:75-84`).
- All 140 local task-level success rates are null. Stored family percentages are correctly labeled aggregate source-screening evidence, not task measurements (`README.md:17-24`, `eval_harness/README.md:128`).
- No exact duplicate normalized goal string was found. This is not semantic/path deduplication.

### What is not admission evidence

All 140 tasks remain pending for environment materialization, known-good completion, reset/contamination checks, native verifier fidelity, adversarial verification, repeated task-level success, stability, safety, and final admission. The source audit is explicit that identity validation is narrower than admission (`research/benchmarks/task-candidates/batch-1-source-audit.md:28`). Therefore the inventory is ready for bridge development, **not ready for calibration runs**.

The intended 35–70% inclusive local task-level rule is clear, but its estimator, initial/max repetitions, stopping rule, boundary-overlap classification, and unstable-task exception are not. Those statistical choices must be preregistered for Part 2 rather than improvised during filtering.

The current smoke is source-balanced—5 REAL, 5 WebArena, 4 WorkArena, 3 Visual, 3 WARC—and all but one case is state-changing. It spans 11 source/site pairs and 20 recorded interaction classes, but the full inventory has 66 classes. WebArena smoke is GitLab-only; Visual smoke has no shopping task; WARC smoke covers only Alaska/Airbnb. This does not establish the “every principal interaction mechanism” claim in `docs/evaluation/SUITE_PLAN.md:15-19`. Smoke result reuse also requires exact configuration identity, which current result/run metadata cannot attest.

Batch 1 contains zero popup/interference and zero long-horizon-memory coverage (`docs/ROADMAP.md:43-49`). The roadmap requires Batch 2 planning concurrently with calibration, whereas the batch report says to add replacements only after gaps are demonstrated (`research/benchmarks/task-candidates/batch-1-report.md:138-145`). The practical reconciliation is to plan/source candidates concurrently but admit replacements only against measured gaps.

Strong-reference filtering alone does not prove a useful MiniMax executor distribution. A task with 35–70% success for BrowserGym GenericAgent + GPT-5.6 Sol Ultra may be near-zero for MiniMax-M3. Run the exact MiniMax seed baseline before freezing the final suite; do not use its noisy first pass as reference calibration.

After ADR-0003, a minimal structured seed can be implemented without violating the component contracts **if** the selected backend supports the session interface, accessibility/DOM observations, epoch invalidation, required actions/tabs, and canonical tracing. The contracts preserve backend neutrality, but they are not complete enough for a real bridge today: lifecycle, errors/cancellation, profiles/auth, frames, native input, dialog/popup/file transfer, reset integration, and trace/artifact ownership need concrete semantics.

Optimizer read-only infrastructure is correct for experiment validity. Backend or lane experiments therefore cannot be in-iteration optimizer edits: a conductor must create a new pinned substrate/lane campaign, record its full identity, rerun environment/bridge conformance and noise calibration, then compare campaigns on the same frozen suite. ADR-0015 permits parallel campaigns (`docs/adr/0015-auto-research-loop-architecture.md:80-85`) but does not yet define this substrate-change/recalibration workflow.

The proposed first vertical slice is conceptually small—one source, one task, one structured seed, one native verifier—but not executable. WARC is the likely lowest-dependency candidate because replay can be local, but this is an inference and `NOT VERIFIED` until the WACZ, replay stack, and verifier are obtained.

### Trace and artifact enforcement map

| Requirement | State | Evidence |
|---|---|---|
| Append-only `trace.jsonl` per task run | `documented-only` | ADR path/layout is explicit (`docs/adr/0004-trace-storage.md:17-19`); no writer exists. |
| Full event schema | Schema-specified, not runtime-enforced | Schema requires 12 load-bearing fields (`schemas/trace-event.schema.json:6-18`); consumer checks only seven (`judge_harness/src/opti_judge/evidence.py:20-28`, `judge_harness/src/opti_judge/evidence.py:88-95`). |

| Timestamps/monotonic clock/epoch | Timestamp/clock schema-only; epoch optional/untyped by event | No validator checks chronological/monotonic order. `browser_state_epoch` is optional (`schemas/trace-event.schema.json:93-99`). |

| Safe resolution and result linkage | Missing / incompatible | Generic result requires run ID and an artifact array (`schemas/result.schema.json:6-15`, `schemas/result.schema.json:82-87`); `TaskResult` emits no run ID and uses a string map (`eval_harness/src/opti_eval/models.py:33-46`). No safe URI/path resolver exists. |
| Redaction/export and retention | `documented-only` | Export redaction is required and holdout must use a separate store (`docs/adr/0004-trace-storage.md:29-35`); sizing/retention remains open (`docs/adr/0004-trace-storage.md:47-49`). |
| Final-state/action/network/side-effect sufficiency | Impossible to validate | Payload is unconstrained (`schemas/trace-event.schema.json:100-103`), there is no terminal-evidence requirement, and missing trace is currently skipped by eligibility. |

The first real bridge needs one shared, fail-closed trace-bundle validator at the evaluation boundary: require a trace for every terminal task result; apply the full schema; bind exactly one expected run/task/config identity; preserve and validate append order, unique IDs, contiguous monotonic sequences/times/epochs; require terminal verifier/final-state evidence; verify CAS hash/media/visibility and path safety; and link the validated bundle into the result. Consumers may then filter visibility, but must not be the first integrity validator.

### Feasibility and capacity

These are order-of-magnitude planning estimates, not measured runtime claims. Manifest timeouts total about 18,300 seconds (5.1 task-hours) for smoke and 129,000 seconds (35.8 task-hours) for one 140-task pass. With four workers, ideal lower bounds are roughly 1.3 and 9 hours; practical windows are about **1–5 hours for smoke** and **10–36 hours for one all-140 pass**, with WorkArena serialized and resets/rate limits adding delay. Three calibration trials per task are 420 episodes, about **108 timeout task-hours** and roughly **30–108 wall hours**. Five MiniMax repeats over an approximately 100-task suite add about 500 episodes and another order of 36–128 wall hours.

Four LLM judge roles need at least 25 distinct, class-balanced trusted cases each, so the floor is about 100 calls and the realistic corpus is hundreds once family/class stratification and disagreements are included (`judge_harness/README.md:16-19`, `docs/BRINGUP_RUNBOOK.md:117-126`). Per-task interpretation of the six probe cases is up to 840 task-trajectory cases and likely `10^2` operator-hours. Environment provisioning is days to weeks once access exists; ServiceNow/REAL approval can impose unbounded calendar latency.

Storage is wholly unmeasured. As a planning sensitivity, 50–500 MB per episode would make three all-140 calibration repeats about **21–210 GB** and five 100-task noise sweeps about **25–250 GB**, before retained videos/native traces or campaigns. This can exceed the runbook's ~150 GB host assumption (`docs/BRINGUP_RUNBOOK.md:38-48`). Measure p50/p95 bytes, tokens, latency, and cost during the first five-source smoke before committing to retention or campaign budgets.

The documented `$10` per iteration and `$500` per 25 iterations (`docs/BRINGUP_RUNBOOK.md:154-158`) are not yet supported by measured episode/model/judge costs. A full baseline/treatment cycle is at least hundreds of executor episodes, so the budget may be incompatible with the intended gate even before judge costs.

The critical long pole is B1+B2: access to five environments plus deterministic resets, real bridges, and per-task verifier evidence—especially the gated WorkArena/REAL surfaces. The most likely visible bring-up failure is unavailable/nondeterministic environment state or an invocation mismatch. The more dangerous failure is invisible: the current fail-open result/trace path can label unsupported output trustworthy.

## 9. Dependency-ordered path

The shortest valid path preserves the runbook's intent while removing its circularities:

1. **Make evidence fail closed first.** Fix FOUND-001/002; settle the calibration sampling rule; define the private holdout source/commitment contract; add source-specific environment and identity lock schemas.
2. **Resolve only the decisions needed for a seed.** Decide ADR-0003 and clarify ADR-0002's relationship to the accepted component tree. Implement exact model invocation/identity and the smallest structured seed/session/tracer surface. This can proceed in parallel with environment access work.
3. **Provision the trusted host.** Create the two users/store boundary, wire secrets/auth, verify every model role, and prove optimizer confinement.
4. **Provision/reset environments.** Pin each runtime and native verifier; implement command reset plus observable-state hash; reproduce two identical resets. Enforce WorkArena serialization.
5. **Build one strict bridge at a time.** Start with the lowest-dependency source proven available (likely WARC, but `NOT VERIFIED`), then each remaining family. Every bridge must bind full identity, native result, trace, terminal evidence, and CAS artifacts.
6. **Reach real five-source smoke.** For one task/source, run known-good, six verifier probes, reset/contamination checks, real reference and seed execution, and T1. This is the `R2` checkpoint.
7. **Admit and calibrate all 140.** Materialize every task; audit oracle/reset/verifier/safety/duplicates; run preregistered repeated reference calibration **and an exact MiniMax seed baseline**; store a disposition for every task.
8. **Fill measured coverage and freeze.** Source Batch 2 in parallel, admit only justified replacements, freeze primary/smoke/regression, get owner decisions on ADR-0008/0009, and bind every artifact to the frozen identity.
9. **Create/seal the separate holdout before optimization.** Commit the private disjoint pool in the conductor store and prove physical/semantic non-disclosure.
10. **Measure exact-seed noise.** Run ≥5 real repeats only after suite, executor, harness, browser, environment, verifier, and config are frozen; derive thresholds and budgets.
11. **Calibrate dependent analysis.** Wire the four real judge roles and Analyst on real traces/corpora; meet per-family operating points.
12. **Rehearse and authorize.** Execute the real injection catalog under confinement, resolve failures, obtain owner authorization, then begin safe iteration 1.

Milestones: **HEAD → steps 1–6: one real smoke/source → step 7: all-140 calibration → steps 8–10: frozen suite/noise → steps 11–12: safe iteration 1.**

## 11. Direct answers

### What can be truthfully run today?

Catalog/suite/document validation, fixture execution over all 140 or smoke-20, unit-tested loop/judge plumbing, and synthetic command/registry adapter experiments. None is a real browser benchmark. No source bridge, environment, browser seed, native verifier path, or model call was verified.

### What is the first missing executable vertical slice?

One pinned real task—likely a WARC smoke case if its assets prove available—with a reproducible environment/reset, exact resolver, minimal structured session/seed (or pinned reference harness for calibration), native verifier, six admitted task-level probe outcomes, canonical append-only trace, hashed CAS artifacts, and strict full-identity result validation. It must demonstrate both a valid success and fail-closed infrastructure/evaluator failures.

### Which prerequisite is the long pole?

Five-source environment/bridge bring-up with deterministic resets and admitted native verifiers, especially gated WorkArena/REAL access and manual per-task probe evidence. Model invocation and backend decisions can be worked in parallel; they do not remove this dependency.

### Are the 140 tasks ready for calibration?

**No.** Their identities, provenance, counts, locks, and provisional suite membership are ready for implementation work. Their runtime materialization, resets, known-good paths, verifier fidelity, probe admissions, safety/duplication review, and task-level measurements are all pending.

### What minimum evidence would change this part's verdict?

To change `R1` to `R2`: close FOUND-001/002; reproduce a clean-host deployment and two identical resets for every source; produce one real smoke result/source with full identity, admitted native verifier, schema-valid single-run trace, terminal state, safe CAS artifacts, and correct invalid/error mapping; and record a preregistered calibration rule plus private holdout source contract.

To claim `R3`: add complete per-task environment/oracle/probe admissions; all-140 identity-bound reference results and MiniMax seed baseline with explicit dispositions; frozen coverage-justified suites; a sealed disjoint holdout; non-synthetic noise on the exact seed identity; calibrated judges/Analyst; successful real injection rehearsal; and owner authorization.
